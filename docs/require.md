# 要件定義

## 1. 目的

本アプリは、`0~9` の数字カードと四則演算を用いて、ある数値を `0` に到達させることを基本理念とするボードゲームを、Webブラウザ上でリアルタイムに遊べるようにすることを目的とする。

本アプリは単なる `basic` ルールの実装に留まらず、以下を継続的に行える構造を前提とする。

- 戦略カードの追加、削除、効果調整
- バグカードの追加、削除、効果調整
- 勝利条件、フェーズ、特殊イベントの調整
- 拡張ルールセットの追加
- 既存ルールとは異なる新ゲームモードの追加

## 2. スコープ

初期スコープは、`rules/basic.yaml` をもとにしたオンライン対戦アプリの要件整理とする。

対象範囲:

- プレイヤーがルームを作成し、参加できること
- 複数人で同一ゲーム状態をリアルタイム共有できること
- 通常ゲーム、決戦フェーズ、レイド戦を進行できること
- 戦略カード、バグカード、残留効果を扱えること
- 将来的なルール差し替えや拡張をしやすいこと

対象外:

- 課金機能
- レーティング
- マッチメイキング
- AI対戦
- 多言語対応

## 3. 技術前提

- ランタイム: Cloudflare Workers
- サーバフレームワーク: Hono
- フロントエンド: React
- 通信方式: WebSocket

補足:

- Cloudflare Workers 上で長時間の双方向接続を扱うため、接続管理とゲーム状態管理の責務を明確に分離する
- 永続化やルーム状態保持には、将来的に Durable Objects の採用を前提とした構成を取りやすい設計とする

## 4. 基本理念

本ゲームシステムの中核は以下とする。

- 数字 `0~9` と四則演算を使って目標値を `0` にする
- 単純な加減算だけでなく、ルール効果によって局面が変化する
- プレイヤー間の妨害、介入、協力、裏切りが発生しうる
- ルールは固定コードではなく、差し替え可能な定義として扱う

このため、アプリケーション設計上は以下を重要視する。

- ルール記述と進行エンジンの分離
- UI とゲームロジックの分離
- 通信プロトコルとゲームドメインの分離
- 1つのルールセットに閉じないデータ駆動設計

## 5. 利用者とユースケース

### 5.1 利用者

- ルーム作成者
- 参加プレイヤー
- 観戦者
- 将来的なルール作成者、運営者

### 5.2 主要ユースケース

- プレイヤーがルームを作成する
- 他プレイヤーがルームに参加する
- 参加者が準備完了後にゲームを開始する
- 各プレイヤーが手番で行動する
- 全参加者の画面に同一のゲーム状態が同期表示される
- ルールに従ってフェーズ遷移する
- セッション終了後に結果を確認する
- 将来的に別ルールセットを選択して開始する

## 6. 機能要件

### 6.1 ルーム管理

- ルームを作成できること
- ルームIDで参加できること
- 参加人数の上限を設定できること
- ルーム内でプレイヤー一覧を確認できること
- ホストがゲーム開始操作を行えること
- 未開始状態で戦略やルール設定を確認できること

### 6.2 セッション管理

- セッション開始前にルールセットを選択できること
- プレイヤーごとの勝利数を管理できること
- セッション単位でゲーム数を管理できること
- セッション勝者を判定できること
- セッション終了後に再戦または解散を選べること

### 6.3 ゲーム進行

- 山札、手札、場札、除外札を管理できること
- 手番順を決定し、順番に行動させられること
- 手番ごとの選択可能行動を判定できること
- セット数、HP、残留効果などの状態を管理できること
- 通常ゲーム、決戦フェーズ、レイド戦を切り替えられること
- 勝敗条件を判定できること
- プレイヤー同士の相談、協力、次ゲーム以降を含むゲーム内完結の交渉を妨げないこと
- ゲーム外でしか履行できない交渉はルール違反として扱えること
- 不正な操作はサーバ側で拒否されること

### 6.4 リアルタイム同期

- ゲーム状態の更新を全参加者へ即時反映できること
- プレイヤーごとに見えてよい情報だけを配信できること
- 接続再開時に最新状態へ復帰できること
- ネットワーク遅延や重複送信があっても状態整合性を保てること

### 6.5 ルールシステム

- ルールセットを識別子付きで複数管理できること
- 戦略カード、バグカード、フェーズ、勝利条件を定義データとして分離できること
- 数値調整をコード変更なしで実施しやすいこと
- ルールごとの使用可能カード、初期設定、進行条件を差し替えられること
- ルール追加時に既存UIや通信仕様への影響を最小化できること

### 6.6 観戦、ログ、履歴

- 観戦者が進行中のゲームを閲覧できること
- 行動履歴を時系列で確認できること
- どの効果で状態が変わったか追跡できること
- 裁定上必要な宣言、交渉、ルール違反指摘を記録または参照できること
- 裁定検証用にイベントログを保持できること

## 7. 非機能要件

### 7.1 拡張性

- 新しい戦略カードやバグカードを追加しても、既存ロジックの大規模改修が不要であること
- 新フェーズや新勝利条件を追加しても、コア進行エンジンが破綻しないこと
- `basic` 以外のルールセットを追加可能であること

### 7.2 保守性

- ゲーム進行ロジックは責務ごとに分離されていること
- 効果判定は個別モジュール化されていること
- ログから状態遷移を追跡できること
- テストしやすい純粋関数ベースのロジックを優先すること

### 7.3 信頼性

- 正当な手順以外ではゲーム状態が壊れないこと
- WebSocket切断時にも復旧可能であること
- 二重送信や古いメッセージにより状態が重複更新されないこと

### 7.4 性能

- 4人対戦を基準として、手番操作後の画面同期が体感上即時であること
- ルーム数増加時にもゲームごとの状態管理が独立していること

## 8. 設計方針

### 8.1 全体方針

- サーバを唯一の正とし、クライアントは表示と入力に徹する
- ルール処理はサーバ側で完結させる
- クライアント側で行う判定は補助表示に限る
- 進行はイベント駆動で実装する

### 8.2 ドメイン分割

最低限、以下の責務に分割する。

- ルーム管理
- 接続管理
- セッション管理
- ゲーム進行管理
- ルール定義管理
- 効果解決
- ログ記録

### 8.3 ルール拡張方針

ルール拡張のしやすさを最優先し、以下を前提とする。

- カード効果を if 文の積み上げで実装しない
- 戦略、バグ、フェーズを識別子ベースで扱う
- 効果は `発動条件` `対象` `処理` `制約` を分けて表現できるようにする
- フェーズ遷移は状態機械として表現できるようにする
- ルール差分で `basic` を継承して一部上書きできる形が望ましい

## 9. データモデル要件

少なくとも以下の概念を独立したデータとして持てること。

- RuleSet
- Session
- Game
- Room
- Player
- Deck
- Hand
- Field
- TurnOrder
- Strategy
- Bug
- Effect
- Phase
- Action
- EventLog

## 10. 通信要件

WebSocket メッセージは、最低限以下の性質を持つこと。

- メッセージ種別が識別できること
- 対象ルーム、対象ゲーム、送信者が識別できること
- クライアント要求とサーバ通知が分離されていること
- 再送や重複検知のための識別子を持てること
- 状態全量同期と差分同期の両方を扱えること

想定メッセージ例:

- ルーム参加
- ルーム退出
- 準備完了
- ゲーム開始
- 行動要求
- 行動結果通知
- フェーズ遷移通知
- 状態同期
- エラー通知

## 11. UI要件

- 現在フェーズが明確に分かること
- 現在のセット数、HP、勝利数、残留バグが見えること
- 手番プレイヤーが明確に分かること
- 実行可能な行動だけが選びやすく表示されること
- 行動結果と効果解決ログが把握できること
- 交渉や相談を行うためのコミュニケーション手段、または禁止される交渉の注意事項を確認できること
- 観戦者表示とプレイヤー表示を分離できること

## 12. テスト要件

- 四則演算裁定を個別に検証できること
- フェーズ遷移を個別に検証できること
- 各カード効果を個別に検証できること
- 代表的なセッション進行をシナリオテストできること
- ゲーム内完結の交渉を許容し、ゲーム外取引に該当する宣言を禁止または記録できることを検証できること
- WebSocket 切断、再接続、重複送信を検証できること
- ルール追加時に既存 `basic` ルールの回帰確認ができること

## 14. 要件定義上の重要原則

- ゲームルールはアプリ本体に埋め込みすぎない
- ルール変更をアプリ全面改修に繋げない
- クライアントの見た目変更とゲームロジック変更を分離する
- `basic` ルールの実装は最初の1例に過ぎず、将来的な派生ルールの土台として設計する
